01 — Comentários

Comente em PT-BR

Todo o time fala português. Comentários em inglês criam barreira de entrada para novos membros e dificultam revisões rápidas em campo.

💬

Explique o PORQUÊ, não o QUÊ

O código já diz o que ele faz. O comentário deve explicar a decisão por trás — a restrição física, o comportamento esperado, o bug que aquilo corrige.

✗ Evitar
// incrementa i
i++;

// calcula distancia
float d = v * t;
✓ Preferir
// Avança índice para não regredir no mapa
// (robô nunca volta à trajetória anterior)
i++;

// Odometria: delta_dist * cos(theta_medio)
// Usa ângulo médio para reduzir erro de integração
float d = v * t;
📌

Comentários de seção em funções longas

Use blocos de comentário para separar etapas dentro de funções grandes. Facilita navegar sem ler tudo.

Exemplo — virtual_line_run()
// --- 1. ODOMETRIA: estima posicao X,Y pelo encoder ---
float delta = distancia - s_ultima_distancia;
...

// --- 2. BUSCA PONTO ALVO no mapa (look-ahead) ---
size_t idx = busca_ponto_lookahead(x, y);
...

// --- 3. CALCULA ALPHA (angulo ao ponto alvo) ---
float alpha = atan2f(dy, dx) - heading_rad;
🚫

Não comente código morto — delete

Código comentado polui, cria dúvida e nunca volta a ser usado. O Git guarda o histórico — use git log se precisar recuperar algo.

✗ Evitar
// set_fan_voltage(3.0);
// delay(100);
// if (modo_antigo) { ... }
nova_funcao();
✓ Preferir
nova_funcao();

// Ventilador removido no hardware v3.
// Ver commit a3f8c2 se precisar reverter.
02 — Variáveis

Nomes que revelam intenção

Uma variável bem nomeada dispensa comentário. Se você precisou comentar o que ela guarda, o nome está errado.

🐍

C++ → snake_case · Dart → camelCase

distancia_percorrida
velocidade_base
s_look_ahead
DistPercorrida
vel
x1
lookAheadDistance
isConnected
robotProvider
LookAhead
flag
tmp
🔒

Prefixos para escopo (C++)

Variáveis estáticas de módulo usam prefixo s_. Constantes de configuração ficam em controllers.h como #define em UPPER_CASE.

Convenção de prefixos
// Variável estática de módulo
static float s_posicao_x = 0.0f;
static bool  s_ativo     = false;

// Constante de configuração
#define VL_LOOK_AHEAD_PADRAO   0.12f
#define PID_KP_PADRAO          1.5f

// Parâmetro de função — sem prefixo
void ajusta_velocidade(float velocidade_alvo);
📐

Inclua a unidade no nome quando relevante

Evita erros de conversão. Se uma variável guarda metros, diga que é metros.

✗ Ambíguo
float distancia;
float angulo;
float tempo;
✓ Claro
float distancia_m;
float angulo_graus;
float tempo_ms;
03 — Funções

Funções com uma responsabilidade

Uma função deve fazer uma coisa, fazer bem, e ter um nome que descreva exatamente o que ela faz.

🏷️

Nome = verbo + substantivo

Funções são ações. O nome deve começar com verbo e deixar claro o que é transformado ou retornado.

✗ Evitar
mapa();
sensor();
dados();
processar();
helper();
✓ Preferir
resetar_mapa();
ler_sensor_frontal();
calcular_odometria();
processar_comando_ble();
enviar_telemetria();
📦

Prefixo de módulo no nome (C++)

Funções públicas de um módulo levam o prefixo do módulo. Facilita autocompletar e deixa clara a origem.

Padrão módulo_ação()
// mapper_module.cpp
void   mapper_iniciar();
void   mapper_atualizar(float dist, float angulo);
void   mapper_suavizar(int janela);
size_t mapper_tamanho_mapa();

// virtual_line.cpp
void  linha_virtual_iniciar();
float linha_virtual_executar(float dist, float heading);
void  linha_virtual_parar();
✂️

Máximo ~40 linhas por função

Se uma função não cabe na tela, provavelmente está fazendo mais de uma coisa. Extraia sub-funções com nomes descritivos.

Regra prática: se você precisou escrever um comentário de seção dentro de uma função (como // --- ETAPA 2 ---), considere extrair aquela etapa para uma função separada.
04 — Arquivos e Módulos

Estrutura previsível

Cada módulo tem exatamente um .h e um .cpp. O .h é a interface pública; o .cpp guarda o estado e a lógica.

📁

Um módulo = uma responsabilidade

Estrutura lib/
lib/
├── controllers/      # Parâmetros PID e configurações
├── virtual_line/     # Pure Pursuit — lógica de seguimento
├── mapper_module/    # Gravação e gerenciamento do mapa
├── giroscopio_module/# Leitura e integração do giroscópio
├── line_sensors/     # Leitura dos sensores de linha
├── bluetooth/        # Comunicação BLE
├── state_machine/    # Máquina de estados do robô
└── user_interface/   # Decode de comandos BLE → ações
🛡️

Header guard em todo .h

Padrão obrigatório
#ifndef NOME_MODULO_H
#define NOME_MODULO_H

#include <Arduino.h>

// Declarações públicas aqui

#endif // NOME_MODULO_H
🔐

Estado interno como static no .cpp

Variáveis de estado de um módulo nunca ficam no .h. Ficam como static no .cpp, invisíveis para o resto do código.

mapper_module.cpp
// Estado privado do módulo — não exportado
static bool  s_mapeando       = false;
static float s_ultima_distancia = 0.0f;
static float s_ultimo_angulo    = 0.0f;

// Mapa compartilhado com virtual_line (extern no .h)
std::vector<TrackPoint> g_mapa_pista;
05 — Git

Histórico que conta uma história

O git log deve funcionar como um diário técnico da equipe. Commits bem escritos valem mais que comentários no código.

📝

Formato: tipo: descrição curta em PT-BR

feat: adiciona perfil de velocidade adaptativo
fix: corrige overflow no índice do mapa
refactor: extrai blend em função separada
docs: atualiza README com fluxo BLE
ajustes
teste
fix bug
WIP
Tipos válidos: feat · fix · refactor · docs · test · chore · perf
🌿

Branches com nome descritivo

feature/salvar-mapa-eeprom
fix/oscilacao-pure-pursuit
experiment/blend-adaptativo
teste2
branch_nova
guilherme
🔬

Um commit = uma mudança lógica

Não agrupe mudanças não relacionadas no mesmo commit. Isso facilita reverter bugs específicos sem perder outras melhorias.

Regra prática: se a mensagem do commit precisar de "e" para descrever o que foi feito ("fix bug e adiciona feature"), separe em dois commits.
Resumo

Checklist rápido

Antes de fazer commit, passe por essa lista.

□  Comentários estão em PT-BR
□  Comentários explicam o PORQUÊ, não o QUÊ
□  Não há código comentado — só código vivo ou deletado
□  Variáveis têm nomes descritivos (sem x1, tmp, flag)
□  Unidades estão no nome quando relevante (_m, _ms, _graus)
□  Funções têm prefixo de módulo (mapper_, linha_virtual_...)
□  Headers têm guard (#ifndef ... #endif)
□  Estado interno é static no .cpp
□  Mensagem de commit segue o padrão tipo: descrição
